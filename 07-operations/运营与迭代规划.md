# 运营与迭代规划

> 阶段：运营与迭代　｜　版本：v1.0　｜　日期：2026-06-16
> 回答「上线后怎么衡量、怎么增长、怎么迭代」。

## 1. 指标体系

### 北极星指标（North Star）
**完成「上传→润色→导出 PDF」的有效用户数**（即真正拿到成品简历的人数）。

### 转化漏斗
```
访问 → 上传简历 → 解析成功 → 发起润色 → 查看对比 → 采纳修改 → 导出 PDF
```
每一步计算转化率，定位流失环节。

### 核心指标
| 维度 | 指标 | 目标（参考） |
|---|---|---|
| 转化 | 上传→导出完成率 | ≥ 60% |
| 质量 | 修改意见采纳率 | ≥ 70% |
| 质量 | 解析成功率 | ≥ 98% |
| 体验 | 润色首字延迟 | 越低越好（受中转站影响） |
| 体验 | 端到端任务成功率 | ≥ 95% |
| 成本 | 单次润色 token 消耗 | 监控并优化 |

## 2. 埋点事件表

| 事件 | 触发时机 | 关键属性 |
|---|---|---|
| `resume_upload` | 上传完成 | 文件类型、大小、是否解析成功 |
| `parse_result` | 解析返回 | 成功/失败、失败原因 |
| `polish_start` | 发起润色 | intent、是否含 JD |
| `polish_first_token` | 首字到达 | 延迟 ms |
| `polish_done` | 流式结束 | 总耗时、token、是否降级 |
| `diff_action` | 采纳/拒绝 | section、accept/reject |
| `export_pdf` | 导出成功 | 采纳条数 |
| `error` | 任一环节报错 | 阶段、错误码 |

> 隐私：埋点不记录简历正文与 PII，仅记录行为与聚合属性。

## 3. 迭代 Roadmap

| 阶段 | 内容 |
|---|---|
| MVP（已交付） | 上传→对齐JD改写+逐条意见→采纳→导出PDF |
| 迭代1 | 历史记录、逐条采纳后重导出、PDF 多模板 |
| 迭代2 | **接入 RAG**：岗位 JD/面经/优秀简历范式知识库（`knowledge.retrieve_context` 实现）；用户不填 JD 时按岗位类别召回 |
| 迭代3 | 批量「一简历多岗位」适配、ATS 关键词覆盖度评分 |
| 迭代4 | 面试模拟 Agent（复用平台 Agent 能力） |

## 4. RAG 升级触发条件
- 当「用户不提供 JD」占比高 → 需按岗位类别召回典型 JD。
- 当需要注入岗位级增值知识（ATS 关键词、能力模型、优秀范式）以提升专业度。
- 升级方式：仅实现 `knowledge.retrieve_context`，主链路与接口零改动（见 SDD §8）。

## 5. 复盘机制
- **节奏**：每个迭代结束做一次复盘。
- **内容**：指标达成情况、漏斗流失点、用户反馈、模型质量抽检（幻觉/事实一致性）、成本。
- **产出**：问题清单 + 下一迭代需求池排序。

## 6. 风险监控
- 模型质量：定期抽检润色结果是否出现幻觉/事实改变。
- 成本：监控 token 与中转站额度，异常突增排查循环请求/超长上下文。
- 可用性：中转站故障时的降级与告警。
