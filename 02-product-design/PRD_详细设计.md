## 1. 总体架构（前后端分离）

```
┌──────────────┐   HTTPS / SSE      ┌─────────────────────────┐        ┌────────────────┐
│   前端 Web    │ ───────────────▶  │      后端 BFF / API       │ ─────▶ │  大模型 API     │
│ (React/Vite) │ ◀─── 流式输出 ──── │  (FastAPI / Node 二选一)  │ ◀───── │ (Claude API)   │
└──────────────┘                    │  ┌────────────────────┐  │        └────────────────┘
                                     │  │ 简历解析服务         │  │
                                     │  │ 意图路由 + 状态机     │  │        ┌────────────────┐
                                     │  │ RAG 检索（JD/面经）   │ ─────▶  │ 向量库 + 对象存储 │
                                     │  │ 润色编排（Prompt）    │  │        └────────────────┘
                                     │  └────────────────────┘  │
                                     └─────────────────────────┘
```

### 1.1 分层职责与数据边界
- **前端**：仅负责交互、上传、流式渲染、对比展示与导出触发。**不直接持有大模型 API Key**，不直接调用模型。
- **后端 BFF**：唯一的对外编排层。负责鉴权、文件解析、RAG 检索、Prompt 组装、调用大模型、流式回传、结果落库。
- **大模型 API 边界**：
  - 仅在后端调用，Key 存于服务端密钥管理（环境变量 / KMS）。
  - 出参/入参限定为「结构化简历字段 + 任务指令」，**不回传任何无关用户隐私字段**。
  - 模型默认选用 `claude-opus-4-8`（高质量润色）/ 可降级 `claude-sonnet-4-6`（成本/时延优化）。
- **存储**：原始文件 → 对象存储（带 TTL）；解析后结构化数据 + 任务记录 → 关系库；RAG 语料 → 向量库。

### 1.2 知识增强策略（直连 + RAG 留缝）
- **本期为直连模式**：目标岗位 JD 由用户在请求时直接提供，是「该岗位要什么」的权威上下文，模型据此做简历↔JD 对齐改写，**核心匹配任务不依赖外部检索**。一套 Prompt + 用户 JD 即可泛化到所有岗位，无需为每个岗位单独建索引。
- **RAG 留缝**：后端在 Prompt 组装处保留知识增强注入点（`knowledge.retrieve_context`，当前返回空）。后续接入 RAG（岗位 JD/面经/优秀简历范式、ATS 关键词、能力模型）时，仅需实现该检索器，主链路与接口零改动。
- **何时升级 RAG**：用户不提供 JD 仅选岗位类别时召回典型 JD；或注入岗位级增值知识把"能改"升级为"改得更懂行"。

## 2. 核心流程

### 2.1 端到端主流程
```
上传文件 → 解析为结构化简历 → 用户确认/补充目标岗位
   → 后端组装 Prompt（简历结构 + JD + RAG 知识）
   → 调用模型（流式）→ 前端实时渲染润色结果
   → 生成「修改对比 + 修改说明」→ 用户采纳/拒绝
   → 导出 PDF → 落库历史记录
```

### 2.2 Agent 状态机（多轮对话流转）
| 状态 | 触发 | 下一状态 | 说明 |
|---|---|---|---|
| `IDLE` | 进入页面 | `UPLOADING` | 等待上传 |
| `UPLOADING` | 文件上传完成 | `PARSING` | — |
| `PARSING` | 解析成功 | `NEED_TARGET` | 解析失败 → `PARSE_FAILED` |
| `NEED_TARGET` | 用户提供/跳过 JD | `POLISHING` | JD 可选 |
| `POLISHING` | 模型流式输出中 | `REVIEW` | 中途失败 → `POLISH_FAILED`（可重试） |
| `REVIEW` | 用户采纳/编辑 | `EXPORTING` | — |
| `EXPORTING` | 导出完成 | `DONE` | — |

意图识别路由：在对话入口区分 `润色简历` / `针对岗位优化` / `仅语法修正` 三类意图，路由到不同 Prompt 模板与参数（温度、是否启用 RAG）。

## 3. 功能详细设计

### 3.1 F1 简历上传与解析
- 入参：文件（PDF / DOCX），限制 ≤ 10MB。
- 解析输出结构化 JSON：
```json
{
  "basics": { "name": "", "phone": "", "email": "", "location": "" },
  "education": [{ "school": "", "major": "", "degree": "", "period": "" }],
  "experience": [{ "company": "", "role": "", "period": "", "bullets": ["..."] }],
  "projects": [{ "name": "", "role": "", "period": "", "bullets": ["..."] }],
  "skills": ["..."]
}
```
- 异常处理：扫描件/图片型 PDF → 提示「无法识别文本，请上传文本版简历」；解析置信度低字段 → 标黄让用户确认。

### 3.2 F2 目标岗位输入
- 支持粘贴 JD 全文，或从平台 RAG 库中选择岗位类别。
- JD 文本经清洗后抽取：岗位关键词、硬性技能、加分项，作为润色对齐依据。

### 3.3 F3 AI 润色改写（核心）
- **润色维度**：动词强化、量化成果、STAR 结构化、关键词对齐、去口语化、精简冗余。
- **约束**：
  - 不得编造用户未提供的经历、数字、公司（**防幻觉**：缺数据时用占位符 `[请补充数据]` 而非杜撰）。
  - 保持事实一致，仅做表达层与结构层改写。
- **输出**：流式返回，按简历模块逐段输出，首 Token < 1s。
- **降级**：模型超时/失败时，自动重试 1 次并切换备用模型；仍失败则返回明确错误与「重试」入口。

### 3.4 F4 / F5 修改对比与采纳
- 逐段展示「原文 / 润色后 / 修改说明」三栏。
- 修改说明示例：`将"负责数据处理"改写为"主导清洗与结构化 5w+ 条多源 JD 数据"，强化动词并量化成果，对齐目标岗位"数据处理"关键词`。
- 每段提供：✅ 采纳 / ↩ 保留原文 / ✏ 手动编辑。

### 3.5 F6 导出
- 输出 **PDF**（reportlab + 内置中文字体 STSong-Light），套用单栏专业风标准模板。
- 导出内容为用户最终采纳后的版本。

### 3.6 F7 历史记录
- 列表展示历史任务（文件名、目标岗位、时间、状态），支持重新下载与「再润色」。

## 4. 接口设计（前后端分离）

| 方法 | 路径 | 说明 | 返回 |
|---|---|---|---|
| POST | `/api/resume/upload` | 上传简历文件 | `{ fileId }` |
| POST | `/api/resume/parse` | 解析为结构化 JSON | 结构化简历 |
| POST | `/api/polish` | 发起润色（SSE 流式） | `text/event-stream` |
| GET | `/api/tasks` | 历史任务列表 | 任务数组 |
| GET | `/api/tasks/{id}` | 任务详情 | 任务详情 |
| POST | `/api/export` | 导出 PDF | PDF 文件流 |

**`/api/polish` 流式事件**：`event: meta`（任务元信息）→ 多个 `event: chunk`（文本增量）→ `event: diff`（结构化修改说明）→ `event: done`。

## 5. 非功能需求
- **性能**：首 Token < 1s；单份完整润色 < 30s。
- **稳定性（Prompt 鲁棒性）**：对空简历、超长简历、纯英文/中英混排、乱码、注入攻击（"忽略以上指令"）等极端输入做自动化测试，保证不崩溃、不越权执行、输出结构稳定。
- **安全与隐私**：传输 HTTPS；原始文件 TTL 自动清理（默认 7 天）；模型调用不落用户敏感明文日志；Key 仅服务端持有。
- **可观测**：记录任务时延、成功率、模型 token 消耗、采纳率。

## 6. 验收标准（DoD）
1. 上传 PDF/DOCX → 解析成功率 ≥ 98%，结构化字段正确。
2. 发起润色，首 Token < 1s，流式逐段渲染无卡顿。
3. 润色结果不出现幻觉（缺数据用占位符），事实与原文一致。
4. 修改对比三栏完整，逐条可采纳/拒绝。
5. 导出 PDF 可正常打开、中文显示正确、格式正确。
6. 极端输入测试用例全部通过（不崩溃 / 不越权）。
7. 前后端分离部署，前端无任何模型 Key，调用均经后端编排。

## 7. 里程碑（建议）
| 阶段 | 内容 | 周期 |
|---|---|---|
| M1 | 架构搭建 + 上传解析 + 基础润色闭环 | 第 1~2 周 |
| M2 | JD 对齐 + 修改对比/采纳 + 导出 | 第 3 周 |

## 8. 风险与对策
| 风险 | 对策 |
|---|---|
| 扫描件无法解析 | 明确提示 + 引导上传文本版；后续接入 OCR |
| 模型幻觉编造经历 | Prompt 强约束 + 占位符策略 + 输出校验 |
| 时延不达标 | 流式输出 + 模型降级 + 分段并行润色 |
| 隐私合规 | 文件 TTL、脱敏日志、服务端持 Key |
